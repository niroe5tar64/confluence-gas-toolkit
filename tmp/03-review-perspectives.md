# ログ設計レビュー観点

## 1. 目的との整合性

### 1.1 問題発生時の原因特定
問題が発生した時に、以下が特定できるか確認してください。

#### ケース1: 「Slack通知が来なくなった」
必要なログ情報:
- [ ] ジョブが実行されたのか？（実行スケジュール外か）
- [ ] Confluence APIから正常にデータを取得できたのか？
- [ ] Slack API呼び出しまで到達したのか？
- [ ] Slack API呼び出しで失敗したのか？（ステータスコード、エラーメッセージ）

**設計での充足度:**
```
✓ [INFO] ジョブ開始 → 実行有無判定
✓ [DEBUG] リクエスト送信 + [DEBUG] レスポンス受信 → API成功有無
✓ [DEBUG] メッセージ送信開始 + [INFO] / [ERROR] → Webhook成功有無
✓ [ERROR] → スタックトレース含む
```

#### ケース2: 「一部のページ更新だけ通知されない」
必要なログ情報:
- [ ] 検索CQLが正しいのか？
- [ ] 検索結果には含まれているのか？
- [ ] ペイロード変換でフィルタリングされていないか？
- [ ] Slack送信時にエラーが発生していないか？

**設計での充足度:**
```
△ [DEBUG] CQL生成 → CQL内容は出力されるが、検索結果の件数は？
△ [INFO] 変更検出 → 大まかな件数は出ているが、スキップされた件数は？
✓ [DEBUG] ペイロード変換 → ページごとに出力されるか未定
△ [INFO] メッセージ送信完了 → 何件送信したか不明
```

#### ケース3: 「通知が重複して届いた」
必要なログ情報:
- [ ] 同じタイムスタンプで複数回実行されていないか？
- [ ] ジョブデータ（前回のタイムスタンプ）が正しく保存されているか？
- [ ] 時系列ソートが正しく機能しているか？

**設計での充足度:**
```
△ [DEBUG] ジョブデータ読み込み成功 → データ内容は出力？
✓ [DEBUG] 検索範囲設定 → since の値が出力される
? [INFO] ジョブ完了 → タイムスタンプ保存の成否は？
```

## 2. 運用面での実行可能性

### 2.1 ログを見る人
**質問:**
- [ ] 開発者（自分）だけが見るのか？
- [ ] 運用担当者も見るのか？
- [ ] 1日何回ログをチェックするのか？

**観点:**
- メッセージの用語は相手に理解できるか？
- 開発者向けなら技術用語OK、運用向けなら日本語で明確に
- ログボリュームは許容範囲か？

**例:**
```
✓ 開発者向け: `[DEBUG] ページネーション { page: 2, cumulative: 50, hasNext: true }`
✗ 運用者向け: わからない
✓ 運用者向け: `[INFO] ページ検索: 2ページ目を取得中（累計50件）`
```

### 2.2 どこでログを確認するのか
**確認箇所ごとの特性:**

| 確認箇所 | 特徴 | ログ形式 |
|---------|------|---------|
| **GAS実行ログ** | 実行直後に見える、保持期間短い | 詳細情報OK（DEBUGまで） |
| **Slack通知** | 非同期、見落とし防止 | 簡潔に（エラーのみ） |
| **Google Drive ファイル** | 永続保存、後から検索可能 | 構造化ログ（JSON）推奨 |
| **外部ログシステム** | 分析・アラート可能 | 構造化ログ（JSON）必須 |

**現設計の問題:**
```
△ Slackへはエラーのみ通知（正常系ログは見えない）
△ 構造化ログ（JSON）への対応がない
△ 履歴保存先が明記されていない
```

### 2.3 ログボリュームの評価

**DEBUGレベルを有効にした場合:**
```
ジョブ1回の想定出力行数:

実行頻度: 毎時間
ログレベル: DEBUG

内訳:
- ジョブ開始/終了: 2行
- API呼び出し（初回 + ページネーション）:
  - Confluence: リクエスト + レスポンス + 各ページ = 約 5～10行
  - Slack: リクエスト + レスポンス × 通知数 = 約 2～10行
- その他（変換、ファイルI/O）: 5～20行

合計: 20～50行/ジョブ実行
24時間: 480～1200行/日

評価:
✓ 許容範囲（GASのログは自動削除される）
△ 本番環境では INFO レベルに下げて運用する必要あり
```

## 3. 実装コスト評価

### 3.1 導入の手間
**変更が必要なファイル数:**

```
Phase 1 (Logger基盤):
- src/utils/logger.ts (新規作成) ... 1ファイル
- src/utils/env.ts (LOG_LEVEL読み込み追加) ... 1ファイル
→ 合計 2ファイル

Phase 2 (エラー改善):
- src/use-case/*.ts (3ファイル) × try-catch改善 ... 3ファイル
- src/services/slack/slack-message.ts ... 1ファイル
→ 合計 4ファイル

Phase 3 (API通信):
- src/clients/confluence-client.ts ... 1ファイル
- src/clients/slack-client.ts ... 1ファイル
- src/services/confluence/recent-changes.ts ... 1ファイル
→ 合計 3ファイル

Phase 4 (詳細トレース):
- src/services/** 複数ファイル
→ 合計 5～10ファイル

総計: 15～21ファイル修正
```

**評価:**
```
✓ 段階的導入が可能
✓ Phase 1-2 だけで基本的な問題追跡は可能
△ Phase 3-4 は手間の割に得られる情報が限定的かも
```

### 3.2 保守性

**ログ追加時のルール:**
```
現在の設計での問題:
△ ログを追加する際の命名規則が明記されていない
△ ファイル単位でのcontext名の決定方法が不明確
△ data に何を含めるべきかの基準がない
```

**改善案:**
```
命名規則:
- [DEBUG] = 処理フロー追跡用（関数の入出力）
- [INFO] = 重要なマイルストーン（ジョブ開始/完了、API成功）
- [WARN] = 想定外だが継続可能な状況（デフォルト値使用）
- [ERROR] = 処理停止、復旧不可な状況

context 命名:
- ファイル名 or クラス名を使用
- 例: "ConfluenceClient", "JobExecutionCheck", "SlackException"

data に含める情報:
- 入力値（CQL、ページID）
- 出力値（取得件数、実行結果）
- 処理の判定条件（if分岐の結果など）
- ただし、認証情報やトークンは除外
```

## 4. 抜け漏れチェック（シミュレーション）

### 4.1 想定される障害シナリオと原因特定の可能性

| シナリオ | 原因候補 | ログで特定可能？ |
|---------|--------|-----------------|
| Slack通知が来ない | ジョブが実行されていない | ✓ [INFO] ジョブ開始 |
| | Confluence APIが失敗 | ✓ [ERROR] API呼び出し失敗 |
| | Slack APIが失敗 | ✓ [ERROR] メッセージ送信失敗 |
| | Slack送信自体が失敗 | △ ログ出力できない→別の監視が必要 |
| 一部ページが通知されない | CQL生成エラー | ✓ [DEBUG] CQL生成 |
| | 検索結果に含まれていない | △ 取得件数は出力されるが詳細不明 |
| | ペイロード変換エラー | △ [DEBUG] あるが詳細不明 |
| 通知が重複 | タイムスタンプが保存されていない | △ [DEBUG] 読み込み成功は出力されるが内容不明 |
| | 時刻の重複 | △ [DEBUG] 検索範囲は出力されるが十分？ |
| GAS実行時間超過 | ページネーション無限ループ | ? 現在のログでは判定困難 |
| | Confluence大量データ取得 | ? ページネーションの途中経過が不十分 |

**評価:**
```
◎ = ログだけで原因特定できる
△ = ログで疑わしい箇所がわかるが、確実ではない
✗ = ログでは判定できない

現設計: △が多い
→ 追加すべきログがある
```

### 4.2 追加すべきログの候補

#### A. 現在の設計では不足している情報

```
1. ジョブデータの内容確認
   現在: [DEBUG] ジョブデータ読み込み成功 { fileName }
   不足: 読み込んだタイムスタンプ値が何か
   追加案: data に { fileName, timestamp } を含める

2. 検索結果の詳細
   現在: [INFO] 変更検出 { count }
   不足: 取得したページの詳細（タイトル、更新者など）
   追加案: [DEBUG] で各ページをログ出力、または
           [INFO] で件数だけ出力し、詳細は [DEBUG] で

3. ページネーション終了判定
   現在: [DEBUG] ページネーション { page, cumulative, hasNext }
   不足: なぜループが終了したのか（hasNext=false? API応答の _links.next がない?）
   追加案: hasNext の理由を data に含める

4. タイムスタンプ変換
   現在: [DEBUG] 検索範囲設定 { since }
   不足: 元の値 → 変換後の値の対応が見えない
   追加案: data に { originalValue, convertedValue } を含める
```

#### B. エラー系の不足

```
5. Slack送信失敗時の詳細
   現在: [ERROR] メッセージ送信失敗 { status }
   不足: どのペイロードを送信しようとしていたのか
   追加案: ペイロードの概要（件数、タイトル）を含める

6. 複数rootPageIds での検索
   現在: CQL生成時のログなし
   不足: どのページIDが指定されているのか不明
   追加案: [DEBUG] で rootPageIds の一覧を出力
```

## 6. 具体的な修正案の検討例

### 例: ページネーション処理のログを強化する場合

**現設計:**
```
[DEBUG] ページネーション { page: 2, cumulative: 50, hasNext: true }
```

**改善案1（詳細版）:**
```
[DEBUG] ページネーション開始 { page: 1, limit: 25 }
[DEBUG] ページネーション結果 { page: 1, resultCount: 25, hasNext: true }
[DEBUG] ページネーション開始 { page: 2, limit: 25 }
[DEBUG] ページネーション結果 { page: 2, resultCount: 23, hasNext: false }
[INFO] 検索完了 { totalCount: 48, pages: 2 }
```

**改善案2（簡潔版）:**
```
[DEBUG] 検索開始 { cql, limit: 25 }
[INFO] 検索完了 { totalCount: 48, pages: 2, duration: 1234ms }
```

**どちらを選ぶ？** → 運用方針によって異なる
- 開発初期段階 → 案1（詳細）で問題追跡しやすく
- 安定稼働後 → 案2（簡潔）で情報量を絞る
